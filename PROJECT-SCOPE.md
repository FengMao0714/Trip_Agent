# 项目范围与约束：智能 Agent 旅游助手

> **版本**：v1.0 | **日期**：2026-04-22 | **状态**：待确认

---

## 1. 项目边界

### 1.1 范围内（In Scope）

| 模块 | 具体内容 | 完成标准 |
|------|----------|----------|
| **对话交互** | 自然语言输入 → Agent 理解意图 → 流式输出回复 | 能正确提取目的地、天数、预算、偏好；支持追问 |
| **智能规划** | ReAct Agent 自主调用工具，生成按天排列的结构化行程 | 3 天行程在 30s 内生成，景点真实存在 |
| **RAG 知识库** | pgvector 存储旅游知识，Agent 检索后辅助生成 | 至少覆盖 3 个城市、200+ 条目 |
| **外部工具调用** | 通过 MCP 协议调用高德地图（POI + 路线）和天气 API | 返回真实数据，距离/耗时可验证 |
| **行程微调** | 用户通过对话局部修改行程 | 改一个景点不会导致整个行程重新生成 |
| **前端展示** | 对话 UI + 行程卡片 + 地图打点 | 桌面端双栏布局，移动端可用 |
| **质量评估** | RAGAS 评估脚本，输出忠实度/相关性指标 | 至少对比 2 组实验 |

### 1.2 范围外（Out of Scope）

| 排除项 | 原因 | 后续可扩展性 |
|--------|------|-------------|
| 用户注册/登录/账户体系 | 毕设 Demo 级别，单会话即可 | 后续可接入 NextAuth |
| 酒店/机票/门票预订 | 产品定位为"规划"非"预订" | 可对接 OTA 开放 API |
| 在线支付 | 不涉及交易 | N/A |
| 多语言/国际化 | 仅面向中文用户 | 可通过 i18n 框架扩展 |
| 社交分享/社区 | 非核心功能 | 可加分享链接功能 |
| 原生移动端 App | 仅 Web 端 | 可用 PWA 或 React Native |
| 行程持久化存储 | 会话级别，刷新后丢失 | 可接入数据库持久化 |
| 实时导航 | 仅做路线展示 | 可跳转高德 App 导航 |

### 1.3 灰色地带（需确认）

| 功能 | 建议 | 工作量评估 |
|------|------|-----------|
| **行程导出（Markdown/文本）** | 建议做，工作量小且答辩加分 | 0.5 天 |
| **对话历史持久化（localStorage）** | 建议做，刷新不丢失当前会话 | 0.5 天 |
| **Agent 推理过程可视化** | 建议做，展示 ReAct 的思考链对答辩有加分 | 1 天 |
| **多会话管理（侧边栏历史列表）** | 可不做，MVP 仅支持单会话 | 1.5 天 |

> [!IMPORTANT]
> **建议优先做"Agent 推理过程可视化"**——在对话中展示 Agent 的思考步骤和工具调用过程。这是答辩时最直观的技术亮点，能让老师清晰看到 ReAct 机制在运作。

## 3. 用户认证策略

### 本次毕设：不做认证

| 决策 | 理由 |
|------|------|
| **不实现用户注册/登录** | 毕设评审关注的是 AI Agent 能力，非用户系统 |
| **不实现会话隔离** | 单用户 Demo 场景，无需多租户 |
| **不实现权限控制** | 无管理后台需求 |

### 替代方案

- 使用 `sessionId`（UUID）标识当前会话，存储在浏览器 `localStorage`
- 后端根据 `sessionId` 维护对话上下文
- 演示时每次刷新或手动新建会话即可

### 论文中如何说明

> "本系统为原型验证阶段，采用基于 SessionID 的会话管理机制，暂未引入用户认证体系。该设计可通过集成 OAuth 2.0 或 NextAuth.js 等方案在后续迭代中扩展为多用户系统。"

## 4. 非功能需求指标

### 4.1 性能目标

| 指标 | 目标值 | 测量方式 | 优先级 |
|------|--------|----------|--------|
| **后端 P95 响应时间** | < 3s（首 Token） | 从收到请求到 SSE 首事件推送 | P0 |
| **完整行程生成** | < 30s（3 天行程） | 从请求到 SSE `[DONE]` 事件 | P0 |
| **单次工具调用** | < 2s | 高德 API / RAG 检索单次耗时 | P0 |
| **前端 FCP** | < 2s | Lighthouse First Contentful Paint | P1 |
| **前端 LCP** | < 3s | Lighthouse Largest Contentful Paint | P1 |
| **前端 TTI** | < 4s | Lighthouse Time to Interactive | P1 |

### 4.2 开发环境 vs 生产环境

| 维度 | 开发环境 | 生产/演示环境 |
|------|----------|-------------|
| **前端** | `npm run dev`（热更新） | `npm run build` + `npm start` 或 Vercel |
| **后端** | `uvicorn --reload`（自动重载） | `uvicorn --workers 2`（多 Worker） |
| **数据库** | Docker 本地 PostgreSQL | 同（Docker） |
| **Redis** | Docker 本地 | 同（Docker） |
| **LLM API** | DeepSeek（低频调用，省钱） | DeepSeek（相同） |
| **日志级别** | DEBUG | INFO |
| **CORS** | 允许 `localhost:3000` | 允许特定域名 |
| **环境变量** | `.env.local` | `.env.production` 或 Docker secrets |
| **编排方式** | 各服务手动启动 | `docker-compose up -d` 一键启动 |

### 4.3 已知技术债与优化方向

| 技术债 | 当前状态 | 影响 | 后续优化方向 |
|--------|----------|------|-------------|
| **无用户认证** | 单会话模式 | 无法多用户隔离 | 接入 NextAuth + JWT |
| **无行程持久化** | 数据仅存内存/前端状态 | 刷新即丢失 | 行程存入 PostgreSQL |
| **RAG 知识库规模小** | 3 城市 200+ 条目 | 覆盖面有限 | 爬虫 + 自动化数据入库 |
| **无自动化测试** | 手动测试为主 | 回归风险 | 加入 pytest + Playwright |
| **无请求限流** | 依赖前端自律 | Token 消耗不可控 | FastAPI 中间件限流 |
| **无监控告警** | 无 | 线上问题无法感知 | 接入 Prometheus + Grafana |
| **地图组件无懒加载** | 页面加载即初始化 | 首屏性能影响 | `dynamic(() => import(...), { ssr: false })` |
| **SSE 断线无重连** | 断开后不恢复 | 长时间生成可能中断 | 加入自动重连 + 断点续传逻辑 |

## 5. 毕设答辩关注点

### 5.1 老师大概率会关注的方面

基于开题报告中的技术关键词和研究方向，预判答辩老师关注点：

| 关注领域 | 可能被问到的问题 | 准备策略 |
|----------|-----------------|----------|
| **Agent 架构** | "你的 Agent 是怎么工作的？ReAct 循环具体怎么实现的？" | 准备 LangGraph 状态机图 + 一次完整推理的日志截图 |
| **与传统方案的区别** | "你这个和携程/马蜂窝有什么区别？" | 强调"智能规划 vs 信息检索"，展示 Agent 自主决策过程 |

### 5.3 推荐 Demo 场景（3 个）

#### 场景 1：标准流程演示（必做）

```
用户输入："我想去北京玩3天，预算3000元，喜欢历史文化，不吃辣"

期望展示：
  → Agent 思考过程可见
  → 调用高德搜索北京历史文化景点
  → 调用天气 API 查询天气
  → 生成 3 天行程（含故宫、天坛、颐和园等）
  → 行程卡片展示每日安排
  → 地图标注所有景点 + 路线
  → 总预算在 3000 元以内
```

#### 场景 2：行程微调演示（重要）

```
在场景 1 基础上继续对话：
用户输入："第二天下午太累了，把长城换成附近的咖啡馆休息"

期望展示：
  → Agent 理解"第二天下午"+"换成咖啡馆"
  → 仅修改第二天下午的活动
  → 其余行程不变
  → 地图更新标注
```

#### 场景 3：复杂约束演示（加分）

```
用户输入："我带 70 岁的爷爷去成都玩 4 天，预算 5000，
         爷爷走不了太远，每天最多 2 个景点，
         要吃清淡的川菜，住在市中心"

期望展示：
  → Agent 提取多重约束（老人、步行限制、景点数量限制、饮食、住宿位置）
  → 生成的行程节奏舒缓，无高强度步行
  → 餐厅推荐清淡川菜而非麻辣火锅
  → 住宿建议在市中心区域
```

### 5.5 答辩风险预案

| 风险 | 概率 | 应对方案 |

## 6. 里程碑与验收节点

| 里程碑 | 目标日期 | 交付物 | 验收标准 |
|--------|----------|--------|----------|
| **M1：技术验证** | 第 2 周末 | Spike Demo | DeepSeek + LangGraph 单 Agent 能输出结构化行程；MCP 能调通高德 |
| **M2：后端核心** | 第 5 周末 | 后端 API | ReAct Agent + RAG + 高德工具链完整可用 |
| **M3：前端 MVP** | 第 8 周末 | 可交互 Web 页面 | 对话 + 行程卡片 + 地图 三大功能联通 |
| **M4：系统联调** | 第 10 周末 | 完整 Demo | 3 个 Demo 场景全部可演示 |
| **M5：论文完成** | 第 12 周末 | 毕业论文 + PPT | 论文定稿 + RAGAS 实验数据 + 答辩 PPT |

> [!WARNING]
> **M1 是最关键的里程碑**——如果第 2 周末技术验证不通过（MCP 调不通或 Agent 输出质量太差），需要立即调整技术方案（如降级为传统 API 封装），不要拖到后面。
