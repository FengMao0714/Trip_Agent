> **版本**：v1.0 &nbsp;|&nbsp; **作者**：冯茂 &nbsp;|&nbsp; **日期**：2026-04-19 &nbsp;|&nbsp; **状态**：草稿

## 1. 产品概述

**一句话描述**：一个基于多智能体协作（Multi-Agent）的 Web 端智能旅游助手，面向有自由行规划需求的普通用户，通过自然语言对话理解用户的预算、偏好和时间约束，自动生成包含食、住、行、游的个性化行程方案，并支持交互式微调，解决传统旅游平台"只能订、不会规划"的痛点。

### 目标用户

| 用户类型 | 占比（预估） | 核心诉求 |
|----------|-------------|----------|
| 自由行新手 | 60% | 不知道去哪、怎么安排，希望一键生成靠谱行程 |
| 有经验但懒得排计划的旅行者 | 30% | 有大致想法，但不想花时间查攻略、算路线 |
| 有特殊需求的定制用户 | 10% | 饮食禁忌、预算严格、亲子/老人同行等复杂约束 |

### 产品定位边界

| 我们是 | 我们不是 |
|--------|----------|
| 智能行程**规划**助手 | OTA 预订平台（不做酒店/机票预订） |
| 对话式 AI 工具 | 传统表单式搜索引擎 |
| Web 端单用户工具 | 社交/社区平台 |
| 毕业设计 Demo 级产品 | 商业化生产系统 |

## 2. 用户故事

### P0 — 必须实现（核心路径）

| 编号 | 用户故事 | 验收要点 |
|------|----------|----------|
| US-01 | 作为**普通游客**，我希望用自然语言描述我的旅行需求（目的地、天数、预算、偏好），以便系统能理解我想要什么样的旅行 | 系统能从自由文本中提取出目的地、天数、预算、旅行风格、饮食偏好等关键信息 |
| US-02 | 作为**普通游客**，我希望系统自动生成一份按天排列的完整行程方案，以便我不用自己查攻略也能获得合理的旅行计划 | 行程包含每日的景点、餐饮、住宿建议，时间安排合理，景点间距离可达 |
| US-03 | 作为**普通游客**，我希望行程中的景点之间有真实的路线和耗时信息，以便我知道实际交通是否可行 | 系统调用地图 API 返回真实距离和预计耗时，而非 LLM 编造的数据 |
| US-04 | 作为**普通游客**，我希望看到行程的流式输出过程，以便在等待时不会觉得系统卡住了 | 行程内容以流式方式逐步呈现在界面上，有加载状态指示 |
| US-05 | 作为**普通游客**，我希望在一个可视化的 Web 界面上与系统交互，以便使用体验友好直观 | 提供对话式 UI，支持消息输入、行程展示、基本交互 |

### P1 — 重要（增强体验）

| 编号 | 用户故事 | 验收要点 |
|------|----------|----------|
| US-06 | 作为**行程微调用户**，我希望通过对话告诉系统"把第二天的故宫换成颐和园"，以便局部修改行程而不用重新生成 | 系统理解修改意图，仅调整指定部分，其余行程保持不变 |
| US-07 | 作为**普通游客**，我希望在地图上看到行程中所有景点的标注和路线连线，以便直观了解空间分布 | 前端集成高德地图，展示景点打点和路线 |
| US-08 | 作为**深度定制用户**，我希望系统在规划时考虑我的预算上限，以便生成的行程花费在我的承受范围内 | 行程中的消费建议不超过用户设定的预算，有费用估算 |
| US-09 | 作为**普通游客**，我希望系统告诉我目的地未来几天的天气，以便我调整行程安排（如雨天去室内景点） | 行程中标注天气信息，恶劣天气时有替代方案建议 |
| US-10 | 作为**普通游客**，我希望系统推荐的景点和餐厅是真实存在的，以便我的行程具有可执行性 | 景点/餐厅信息来自 RAG 知识库或地图 POI 搜索，非 LLM 幻觉 |

### P2 — Nice to Have（锦上添花）

| 编号 | 用户故事 | 验收要点 |
|------|----------|----------|
| US-11 | 作为**普通游客**，我希望系统能记住我之前的对话上下文，以便进行多轮对话优化行程 | 在同一会话内，系统保持上下文一致性（至少支持 10 轮对话） |
| US-12 | 作为**普通游客**，我希望高频查询（如热门城市行程）能快速返回结果，以便不用每次都等 Agent 重新规划 | Redis 缓存命中时，响应时间显著缩短 |
| US-13 | 作为**系统管理员**，我希望能通过 RAGAS 指标评估系统回答质量，以便持续优化 Agent 效果 | 提供评估脚本，输出忠实度、相关性等量化指标 |
| US-14 | 作为**普通游客**，我希望能将生成的行程导出为文本或图片，以便分享给同行伙伴 | 提供行程的纯文本/Markdown 导出功能 |

## 3. 功能清单

### 3.1 对话交互模块

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| 自然语言输入 | 用户通过文本框输入旅行需求 | P0 | 支持中文自由文本输入，长度不限 |
| 意图识别与信息提取 | LLM 从用户输入中提取结构化信息 | P0 | 准确提取目的地、天数、预算、偏好等字段，缺失字段主动追问 |
| 流式输出 | Agent 的回复以 SSE 流式方式推送到前端 | P0 | 首字符延迟 < 2s，输出过程流畅无卡顿 |
| 多轮对话 | 在同一会话中保持上下文 | P2 | 至少支持 10 轮连续对话，上下文不丢失 |
| 追问与澄清 | 信息不完整时主动向用户提问 | P1 | 当缺少必要信息（如目的地/天数）时，系统主动追问 |

### 3.2 智能规划引擎

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| ReAct 推理循环 | Agent 通过"思考-行动-观察"循环完成规划 | P0 | 能自主决定何时调用工具、何时生成回复，循环次数上限 10 次 |
| 用户画像构建 | 从对话中提取并维护用户偏好 | P0 | 偏好信息在会话内持久化，影响后续行程生成 |
| 行程结构化输出 | 生成按天/按时段排列的结构化行程 | P0 | 输出 JSON 格式行程，包含日期、时段、地点、活动、交通方式、费用估算 |
| 预算约束 | 行程规划中考虑预算上限 | P1 | 总费用估算不超过用户预算的 110%（允许 10% 弹性） |
| 行程微调 | 通过对话修改已生成行程的局部内容 | P1 | 支持增加/删除/替换景点、调整顺序，不影响未修改部分 |

### 3.3 知识检索模块（RAG）

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| 知识库构建 | 将旅游数据切片并向量化存入 pgvector | P0 | 至少覆盖 3 个热门城市，每城市不少于 50 个景点/餐厅条目 |
| 混合检索 | 语义检索 + 关键词检索结合 | P1 | 检索 Top-5 结果的相关性（人工评估）>= 80% |
| 知识注入 | 检索结果作为上下文注入 Agent Prompt | P0 | Agent 生成的行程中，景点信息与知识库一致，减少幻觉 |

### 3.4 外部工具调用模块

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| 高德 POI 搜索 | 搜索目的地附近的景点、餐厅、酒店 | P0 | 返回名称、地址、经纬度、评分等信息 |
| 高德路径规划 | 计算两点间的驾车/公交/步行路线 | P0 | 返回距离（km）和预计耗时（分钟） |
| 天气查询 | 查询目的地未来天气 | P1 | 返回未来 3-7 天的天气概况（晴/雨/温度） |
| MCP 协议适配 | 通过 MCP 标准化协议调用上述工具 | P0 | MCP Server 正常运行，Agent 可自主调用；若 MCP 不可用，降级为直接 API 调用 |

### 3.5 前端展示模块

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| 对话界面 | 类 ChatGPT 的对话式 UI | P0 | 支持消息输入、发送、历史消息展示、流式渲染 |
| 行程卡片 | 以卡片形式展示每日行程 | P0 | 行程按天分组，每个景点显示名称、时间、交通方式 |
| 地图展示 | 高德地图嵌入，标注景点位置 | P1 | 景点以 Marker 标注，路线以 Polyline 连接 |
| 响应式布局 | 适配桌面端浏览器 | P0 | 在 1280px 以上分辨率下布局正常 |

### 3.6 系统支撑模块

| 子功能 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| Redis 缓存 | 缓存高频 POI/路线查询结果 | P2 | 缓存命中时响应时间减少 50% 以上 |
| RAGAS 评估 | 对 RAG 回答质量进行定量评估 | P2 | 输出 faithfulness、answer_relevancy 等指标 |
| 日志与监控 | 记录 Agent 调用链和工具调用日志 | P2 | 可追溯每次请求的完整 Agent 推理过程 |

## 4. 数据流图

### 4.1 核心数据流（文字描述）

```
用户输入（自然语言）
    │
    ▼
┌─────────────────────────────────────────────────┐
│  FastAPI 后端                                     │
│                                                   │
│  ① 接收用户消息，传入 LangGraph Agent             │
│     │                                             │
│     ▼                                             │
│  ② Agent（DeepSeek-V3）进入 ReAct 循环            │
│     │                                             │
│     ├──思考──▶ 分析用户意图，决定下一步行动        │
│     │                                             │
│     ├──行动──▶ 选择并调用工具（以下任选）          │
│     │          ├─ RAG 检索（pgvector）             │
│     │          │   └─▶ 返回景点/餐厅知识片段      │
│     │          ├─ 高德 POI 搜索（MCP Server）      │
│     │          │   └─▶ 返回真实 POI 数据          │
│     │          ├─ 高德路径规划（MCP Server）        │
│     │          │   └─▶ 返回距离与耗时             │
│     │          └─ 天气查询（MCP Server）           │
│     │              └─▶ 返回天气预报               │
│     │                                             │
│     ├──观察──▶ 整合工具返回结果                    │
│     │                                             │
│     └──循环──▶ 重复思考-行动-观察，直至行程完成    │
│                 （最大 10 次迭代）                 │
│     │                                             │
│     ▼                                             │
│  ③ 生成结构化行程（JSON）+ 自然语言描述           │
│     │                                             │
│     ▼                                             │
│  ④ 通过 SSE 流式推送至前端                        │
│                                                   │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Next.js 前端                                     │
│                                                   │
│  ⑤ 流式渲染对话内容                               │
│  ⑥ 解析行程 JSON，渲染行程卡片                    │
│  ⑦ 将景点经纬度标注到高德地图上                    │
│  ⑧ 用户可继续对话进行行程微调 → 回到 ①            │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 4.2 数据实体关系

```
用户会话（Session）
  ├── 用户画像（UserProfile）
  │     ├── 目的地（destination）
  │     ├── 天数（days）
  │     ├── 预算（budget）
  │     ├── 旅行风格（style）
  │     └── 饮食偏好（diet_preference）
  │
  └── 行程方案（Itinerary）
        ├── 行程日（DayPlan）[1..N]
        │     ├── 日期（date）
        │     ├── 天气（weather）
        │     └── 活动（Activity）[1..M]
        │           ├── 时间段（time_slot）
        │           ├── 地点名称（place_name）
        │           ├── 地点类型（景点/餐厅/住宿）
        │           ├── 经纬度（lng, lat）
        │           ├── 活动描述（description）
        │           ├── 预计费用（cost）
        │           └── 交通信息（transport）
        │                 ├── 方式（mode）
        │                 ├── 距离（distance_km）
        │                 └── 耗时（duration_min）
        │
        └── 费用汇总（total_cost）
```

## 5. 非功能需求

### 5.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| **首字符延迟（TTFT）** | < 2 秒 | 用户发送消息后，2 秒内开始看到流式输出 |
| **完整行程生成时间** | < 30 秒 | 3 天行程从请求到完整输出不超过 30 秒 |
| **单次工具调用延迟** | < 3 秒 | 高德 API / RAG 检索单次调用耗时上限 |
| **页面首屏加载** | < 3 秒 | Next.js 页面首次加载时间 |

### 5.2 并发与容量

| 指标 | 要求 | 说明 |
|------|------|------|
| **并发用户数** | 1-5 人 | 毕业设计 Demo 级别，不考虑高并发场景 |
| **知识库规模** | 3-5 个城市，200+ 条目 | 覆盖北京、上海、成都等热门目的地 |
| **单次对话轮次** | 最多 20 轮 | 超过后建议用户新建会话 |
| **Agent 最大迭代** | 10 次 | ReAct 循环上限，超过则返回当前最佳结果 |

### 5.3 部署环境

| 组件 | 技术选型 | 部署方式 |
|------|----------|----------|
| **前端** | Next.js (React) | 本地 `npm run dev` 或 Vercel |
| **后端** | Python FastAPI | 本地运行 或 Docker |
| **LLM** | DeepSeek-V3 | 云端 API 调用 |
| **向量数据库** | PostgreSQL + pgvector | Docker 容器 |
| **缓存** | Redis | Docker 容器 |
| **MCP Server** | 高德地图 MCP Server | 本地进程 |
| **整体编排** | Docker Compose（推荐） | 一键启动所有服务 |

### 5.4 可靠性与容错

| 场景 | 处理策略 |
|------|----------|
| DeepSeek API 不可用 | 返回友好错误提示："AI 服务暂时不可用，请稍后重试" |
| 高德 API 调用失败 | Agent 跳过路线信息，仅生成景点推荐，标注"路线信息暂不可用" |
| MCP Server 连接失败 | 降级为直接 HTTP API 调用（需提前实现降级方案） |
| Agent 陷入死循环 | 超过 10 次迭代强制终止，返回当前已有结果 |
| RAG 检索无结果 | Agent 基于自身知识生成，但在结果中标注"未经知识库验证" |

### 5.5 安全性

| 要求 | 说明 |
|------|------|
| API Key 保护 | DeepSeek / 高德的 API Key 存储在环境变量中，不硬编码到代码 |
| 输入过滤 | 对用户输入做基本的 Prompt 注入防护 |
| 请求限流 | 单用户每分钟最多 10 次请求，防止 Token 消耗过快 |

## 附录：术语表

| 术语 | 解释 |
|------|------|
| **Agent** | 具备自主推理和工具调用能力的 AI 智能体 |
| **ReAct** | Reasoning + Acting，一种让 LLM 交替进行推理和行动的范式 |
| **RAG** | Retrieval-Augmented Generation，检索增强生成 |
| **MCP** | Model Context Protocol，标准化的 AI 工具调用协议 |
| **LangGraph** | LangChain 生态中用于构建有状态 Agent 工作流的框架 |
| **pgvector** | PostgreSQL 的向量检索扩展 |
| **SSE** | Server-Sent Events，服务端单向推送协议，用于流式输出 |
| **RAGAS** | RAG 系统的自动化评估框架 |
| **POI** | Point of Interest，兴趣点（景点、餐厅等） |
| **TTFT** | Time To First Token，首个 Token 的响应延迟 |
